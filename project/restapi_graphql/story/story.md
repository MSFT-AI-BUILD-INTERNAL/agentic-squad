# REST API vs GraphQL — 프로젝트 컨텍스트

## 프로젝트 현황

### 서비스 개요

| 항목 | 내용 |
|------|------|
| **서비스명** | SquadMart — B2C 이커머스 플랫폼 |
| **팀 규모** | 백엔드 5명, 프론트엔드 4명, 모바일(React Native) 2명 |
| **현재 스택** | Node.js (Express 4) + PostgreSQL + Redis |
| **클라이언트** | React SPA (웹), React Native (iOS/Android), 파트너 API (B2B) |
| **일일 트래픽** | 약 50만 DAU, 피크 시 초당 3,000 요청 |
| **API 엔드포인트 수** | REST 기준 현재 87개 |
| **서비스 단계** | 운영 2년차, 마이크로서비스 전환 진행 중 (모놀리스 → 4개 도메인 서비스) |

### 현재 겪고 있는 문제

1. **Over-fetching** — 상품 목록 API가 상세 정보까지 모두 반환하여 모바일에서 불필요한 데이터 전송 발생 (평균 응답 크기 12KB, 실제 필요 데이터 3KB)
2. **Under-fetching** — 상품 상세 페이지에서 상품 → 리뷰 → 판매자 정보를 3번 호출해야 함 (N+1 API 호출)
3. **버전 관리 부담** — `/v1/products`, `/v2/products` 등 API 버전이 3개 공존, 파트너사별 지원 버전 상이
4. **클라이언트별 요구 불일치** — 웹은 풍부한 데이터, 모바일은 최소 데이터를 원하는데 동일한 엔드포인트 사용 중

## REST API 장단점

### 장점

| 장점 | 설명 | 우리 프로젝트 맥락 |
|------|------|-------------------|
| **단순하고 직관적** | HTTP 메서드(GET/POST/PUT/DELETE)와 URL 구조만으로 API 의미 전달 | 파트너 B2B API는 REST의 직관성이 온보딩 비용을 낮춤 |
| **HTTP 캐싱 활용** | URL 기반으로 CDN, 브라우저 캐싱, Reverse Proxy 캐싱이 자연스럽게 동작 | 상품 목록/카테고리 등 읽기 위주 API에서 Redis + CDN 캐싱으로 응답 시간 50ms 이하 달성 중 |
| **성숙한 생태계** | Swagger/OpenAPI, Postman, API Gateway 등 풍부한 도구 지원 | 현재 Swagger 기반 API 문서 자동 생성, Kong API Gateway 운영 중 |
| **상태 코드 표준화** | 200, 404, 401 등 HTTP 상태 코드로 에러 처리가 명확 | 파트너사 연동 시 표준 상태 코드 기반 에러 핸들링이 안정적 |
| **마이크로서비스 친화적** | 서비스 간 통신에서 경로 기반 라우팅이 단순 | API Gateway에서 `/products/*` → 상품 서비스, `/orders/*` → 주문 서비스로 라우팅 중 |

### 단점

| 단점 | 설명 | 우리 프로젝트 맥락 |
|------|------|-------------------|
| **Over-fetching** | 필요 이상의 데이터를 응답에 포함 | 모바일에서 상품 목록 조회 시 75%가 불필요한 데이터 — 대역폭 낭비 |
| **Under-fetching** | 하나의 화면에 여러 리소스가 필요하면 다중 호출 | 상품 상세 페이지: `GET /products/123` + `GET /products/123/reviews` + `GET /sellers/456` = 3회 호출 |
| **엔드포인트 폭발** | 클라이언트 요구에 맞춰 유사한 엔드포인트가 증가 | `/products`, `/products/summary`, `/products/mobile`, `/products/detail` 등 87개 엔드포인트 관리 부담 |
| **버전 관리 복잡** | 하위 호환성 유지를 위해 여러 버전을 동시 운영 | v1/v2/v3 공존, 각 버전별 테스트·문서 유지보수 비용 증가 |
| **클라이언트 유연성 부족** | 서버가 응답 형태를 결정, 클라이언트는 선택권 없음 | 웹/모바일/파트너가 같은 응답 구조를 강제로 사용 |

## GraphQL 장단점

### 장점

| 장점 | 설명 | 우리 프로젝트 맥락 |
|------|------|-------------------|
| **정확한 데이터 페칭** | 클라이언트가 필요한 필드만 명시하여 요청 | 모바일: `{ products { id, name, price, thumbnail } }` → 응답 3KB (REST 대비 75% 감소) |
| **단일 엔드포인트** | 모든 요청이 `/graphql` 하나로 통합 | 87개 엔드포인트 → 1개로 라우팅 단순화, API Gateway 설정 간소화 |
| **N+1 호출 해소** | 중첩 쿼리로 연관 데이터를 한 번에 조회 | `{ product(id: 123) { name, reviews { text }, seller { name } } }` = 1회 호출 |
| **강타입 스키마** | SDL로 API 계약이 명확, 자동 문서화·코드 생성 | TypeScript + GraphQL Code Generator로 프론트 타입 자동 생성 가능 |
| **버전 없는 진화** | 필드 추가는 하위 호환, `@deprecated`로 점진적 제거 | v1/v2/v3 통합 → 단일 스키마에서 `@deprecated` 처리로 버전 관리 비용 제거 |
| **클라이언트별 최적화** | 동일 스키마에서 웹/모바일/파트너가 각자 필요한 형태로 요청 | 프론트 4명 + 모바일 2명이 백엔드 변경 없이 독립적으로 화면 개발 가능 |

### 단점

| 단점 | 설명 | 우리 프로젝트 맥락 |
|------|------|-------------------|
| **HTTP 캐싱 불가** | POST 단일 엔드포인트라 CDN/브라우저 캐싱이 동작하지 않음 | 현재 CDN 캐싱으로 달성 중인 50ms 응답 시간을 Apollo Server 캐싱 등으로 대체해야 함 |
| **복잡한 쿼리 보안** | 클라이언트가 깊은 중첩 쿼리를 보내면 서버 부하 급증 | 악의적 쿼리: `{ products { reviews { author { orders { items { product { reviews ... } } } } } } }` → DoS 위험 |
| **학습 곡선** | SDL, Resolver, DataLoader 등 새로운 개념 학습 필요 | 백엔드 5명 중 GraphQL 경험자 1명, 팀 전체 교육 필요 (약 2-3주 예상) |
| **파일 업로드 불편** | 멀티파트 업로드가 표준이 아님, 별도 처리 필요 | 상품 이미지 업로드 기능에서 REST 엔드포인트를 병행해야 할 수 있음 |
| **에러 처리 비표준** | HTTP 200으로 응답하되 body에 errors 포함 → 모니터링 도구가 에러를 감지 못함 | 현재 Datadog APM이 HTTP 상태 코드 기반이라 GraphQL 에러 추적을 위한 커스텀 설정 필요 |
| **N+1 쿼리 문제 (서버 측)** | Resolver가 중첩 호출될 때 DB N+1 쿼리 발생 위험 | DataLoader 패턴 도입 필수, 기존 Sequelize ORM과의 통합 작업 필요 |

## 비교 요약

| 기준 | REST API | GraphQL |
|------|----------|---------|
| **데이터 효율성** | ❌ Over/Under-fetching | ✅ 필요한 것만 요청 |
| **캐싱** | ✅ HTTP 표준 캐싱 | ⚠️ 별도 캐싱 전략 필요 |
| **멀티 클라이언트 대응** | ❌ 엔드포인트 증식 | ✅ 단일 스키마, 클라이언트별 쿼리 |
| **API 버전 관리** | ❌ v1/v2/v3 공존 | ✅ 스키마 진화 (@deprecated) |
| **보안** | ✅ 표준화된 인증/인가 | ⚠️ 쿼리 복잡도 제한 필요 |
| **학습 비용** | ✅ 낮음 (HTTP 메서드 기반) | ⚠️ 높음 (SDL, Resolver, DataLoader) |
| **모니터링** | ✅ HTTP 상태 코드 기반 | ⚠️ 커스텀 설정 필요 |
| **파트너 API** | ✅ 직관적 온보딩 | ⚠️ 파트너 학습 비용 발생 |
| **마이크로서비스 통합** | ✅ 경로 기반 라우팅 | ⚠️ Federation/Gateway 추가 필요 |

## 의사결정 포인트

이 프로젝트에서 논의가 필요한 핵심 질문:

1. **전면 전환 vs 하이브리드** — 모든 API를 GraphQL로 전환할 것인가, REST와 병행할 것인가?
2. **파트너 API 처리** — B2B 파트너사에도 GraphQL을 제공할 것인가, REST를 유지할 것인가?
3. **마이그레이션 전략** — 마이크로서비스 전환과 GraphQL 도입을 동시에 진행할 것인가, 순차적으로 할 것인가?
4. **팀 역량** — GraphQL 학습에 투자할 시간적 여유가 있는가? (다음 릴리스 일정: 6주 후)
5. **캐싱 대안** — CDN 캐싱을 포기하고 Apollo Cache + Persisted Queries로 충분한 성능을 낼 수 있는가?
